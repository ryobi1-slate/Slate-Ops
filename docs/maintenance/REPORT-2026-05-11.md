## Executive summary

Slate Ops is currently a hybrid WordPress plugin shell. All `/ops/*` URLs are caught by broad rewrite rules, then route behavior is decided by `slate-ops.php`, `templates/ops-app.php`, and the React bundle.

The newest CS surface is not consistently named. Current code exposes the sidebar label as `CS`, the route as `/ops/cs-dashboard`, the page header as `Operations Dashboard`, the working sub-tab as `Queue`, and the working panel title as `Shop Queue`. Recent PR descriptions describe this area as `CS Workspace`, but current main does not show a `CS Workspace` sub-tab.

The highest drift risk is around CS Dashboard / Queue / legacy CS, because recent PR descriptions, comments, CSS class names, hash aliases, and current visible labels do not all agree.

## Queue, scheduler, and status disagreement risks

- Queue active statuses include `INTAKE`, `NEEDS_SO`, `READY_FOR_BUILD`, `SCHEDULED`, `IN_PROGRESS`, `BLOCKED`, `QC`, and legacy `PENDING_QC`: `includes/class-slate-ops-rest.php:3891-3901`. `Slate_Ops_Statuses::active()` also includes `ON_HOLD`: `includes/class-slate-ops-statuses.php:84-101`. Result: on-hold jobs are active globally but absent from `/cs/queue`.
- Queue UI filter supports `ready`, `inprog`, and `parts` in JS: `assets/js/ops-cs-dashboard.js:526-536`, but current template only renders chips for `all`, `scheduled`, `blocked`, `closeout`, and `unassigned`: `templates/pages/cs-dashboard.php:347-363`. This is not broken, but it is dead branch logic from earlier CS Workspace phases.
- `list_jobs()` orders Tech/React jobs by `queue_priority` then `updated_at`: `includes/class-slate-ops-rest.php:1297-1306`. `/cs/queue` orders by assigned user, `queue_order`, then `queue_priority`: `includes/class-slate-ops-rest.php:3914-3925`. This can make CS ordering and Tech ordering disagree unless the Tech React layer applies its own queue ordering.
- CS queue save can change `assigned_user_id`: `includes/class-slate-ops-rest.php:4033-4038`. `edit_job()` can also change `assigned_user_id`: `includes/class-slate-ops-rest.php:1581-1587`. Assignment updates therefore have multiple write paths.
- Scheduler `schedule_job()` forces status to `QUEUED`: `includes/class-slate-ops-rest.php:49-58` and `includes/class-slate-ops-rest.php:2053-2066`, but installer migration normalizes `QUEUED` to `SCHEDULED`: `includes/class-slate-ops-install.php:699-704`, and status labels treat `QUEUED` as legacy: `includes/class-slate-ops-statuses.php:29-34`.
- Scheduler hold/unhold writes `ON_HOLD` and then returns to `SCHEDULED` or `READY_FOR_BUILD`: `includes/class-slate-ops-rest.php:3368-3423`. CS status transition rules allow broad ON_HOLD transitions: `includes/class-slate-ops-statuses.php:236-253`. This is powerful, but high-risk because scheduler holds and CS holds may not share the same reason fields.
- `release_job()` writes `scheduling_status = READY_FOR_BUILD` without changing primary `status`: `includes/class-slate-ops-rest.php:2087-2144`. `list_jobs()` has a `ready_only` filter that accepts either `scheduling_status` or primary `status`: `includes/class-slate-ops-rest.php:1246-1251`. This dual-state model needs careful documentation.

## High-risk files for future changes

- `assets/js/ops-cs-dashboard.js`: large vanilla state machine for overview, queue, drag/drop, detail editing, partial save, new-job modal, hash aliases. High risk for naming drift and partial-save regressions.
- `templates/pages/cs-dashboard.php`: mixes overview, queue surface, stubs, drawer, new job modal, and initial JSON. High risk for UI label drift and accidental markup/JS ID breakage.
- `assets/css/ops-cs-dashboard.css`: large page-scoped stylesheet with old Workspace comments and responsive overrides that can undo desktop fixes.
- `includes/class-slate-ops-rest.php`: central REST class with jobs, queue, scheduler, time, BOM/pricing, executive, debug, and permissions. High risk due to multiple write paths touching the same job fields.
- `includes/class-slate-ops-install.php`: creates many tables and runs migrations; queue columns are both in CREATE TABLE and idempotent ALTER code.
- `assets/react/app.js`: minified bundle still owns `/ops/cs`, `/ops/tech`, `/ops/schedule`, `/ops/admin`, `/ops/settings`, and root behavior. Do not rewrite casually.
- `slate-ops.php`: asset routing decides React vs server-rendered behavior; small condition changes can load the wrong app on a route.
- `templates/ops-app.php`: shell/template switch, embed mode, access-denied handling; comments already stale.
- `includes/ui/sidebar.php`: visible route naming source for users; currently labels `cs-dashboard` as `CS` and `/ops/cs` as `CS (legacy)`.
- `includes/class-slate-ops-statuses.php`: canonical status rules; changes here affect CS, Tech, scheduler, queue, and dealer-facing labels.
